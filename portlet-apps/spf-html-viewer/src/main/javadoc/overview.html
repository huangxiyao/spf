1. Overview

This is the administration guide and developer's guide for the Shared Portal Framework (SPF) HTML viewer portlet. The HTML viewer portlet is a JSR-286 portlet for displaying an arbitrary content file you provide (called a view file).

  • The view file may contain any HTML, CSS, or JavaScript so long as it is portal-consumable (eg can be aggregated with other content by the portal to constitute the overall page).

  • The view file may also contain any of the special markup tokens for server-side text interpolation, as supported by the SPF portlet utilities' text interpolation engine. Using these special markup tokens, your view file can:

    • permission content based on user authentication or authorization;
    • display user profile attributes (name, email, etc);
    • present URLs for other static resources (images, etc) or portal pages (but not target portlets on those pages with parameters, modes, etc);
    • consolidate content that otherwise would have to be maintained in multiple view files

  • The view file and its static resources (images, etc), if any, may be localized (for example, using the SPF standard localization business process); the portlet will render the versions which best-fit the user's locale.

  • The portal administrator, when creating an instance of the portlet, configures the name of the view file that is rendered by default. This can be overridden per request by specifying an alternate view file through the URL. (The view file itself, and its other static resources if any, are created and edited separately, not online.)

  • The view file, its static resources, and their localized versions may be stored internal to the portlet application or external, on the filesystem of the portlet server, in the portlet resource bundle folder.

    • It is recommended to store them externally, if you anticipate the administrator will need to change those files.
    • Changes to external files take effect without restarts.

  • The portlet is contained in the htmlviewer.war portlet application and may be deployed as a local portlet with the portal, or as a remote portlet accessed by the portal over WSRP. (In Shared Portal Framework, in practice, the portlet will be deployed remotely in the OpenPortal container, and consumed by Vignette Portal.)

1.1. Installation

1.1.1 Installation of the htmlviewer WAR

The HTML viewer portlet is distributed as a complete portlet application, in J2EE WAR format. The WAR filename is htmlviewer.war. The htmlviewer WAR is distributed through Maven 2:

  • The current Maven 2 repository is http://repo1.corp.hp.com/nexus/content/groups/public/com/hp/it/spf/apps/spf-html-viewer/
  • The Maven group ID is com.hp.it.spf.apps
  • The Maven artifact ID is spf-html-viewer

To get the HTML viewer portlet, download the latest released version of htmlviewer.war from the repository.

The HTML viewer is a JSR-286 portlet application, so it needs a JSR-286-compliant portlet container. It can be local to the portal application, or remoted with the support of a WSRP portlet container. Note: For the Shared Portal Framework platform, we will generally use the remoted approach, employing the SPF-integrated OpenPortal portlet container (installation of this container and the application server in which it runs is not covered here). The remainder of these instructions will assume use of the SPF-integrated OpenPortal container.

The deployment procedures for the HTML viewer WAR are generally the same as for any other JSR-286 portlet application. For the SPF-integrated OpenPortal container, general detailed deployment instructions are documented elsewhere in the SPF administration documentation. For your convenience, here are the steps in summary:

  1. Login to the OpenPortal portlet driver console.

  2. On the "Admin" tab of the OpenPortal console, deploy the HTML viewer WAR.

    Important: make sure the WAR filename is htmlviewer.war first, if it is not already so. (This is because at least some versions of OpenPortal require the filename to match the context root declared in the deployment descriptor, which is htmlviewer.) After downloading it above, it may be a different name, so rename it to htmlviewer.war accordingly before deploying it. Also note that if you are updating an existing deployment, you should not undeploy the existing portlet from OpenPortal first.

  3. Login to the application server (WebLogic) console.

  4. Deploy the htmlviewer.war to the desired targets as usual for a Web application. You will find the htmlviewer.war already staged for deployment by OpenPortal.

    Important: you must deploy to WebLogic the version of the WAR which OpenPortal staged, since OpenPortal rewrites some of the deployment descriptor during the OpenPortal deployment step (above).

  5. Return to the OpenPortal console.

  6. On the "Portlets" tab of the OpenPortal console, create your producer or find an existing producer to use. Then publish the HTML viewer portlet to that producer (you should see it listed in the "Unpublished Portlets" list).

    Important: make sure the producer is enabled and has registration required. The latter prepares the system to manage portlet preferences for each instance - a prerequisite for the HTML viewer portlet. If using an existing producer where registration was not required, you can change it - however you will have to delete and re-create any remote portlet server (RPS) and associated portlet instances in Vignette.

  7. Done. You can now create and configure one or more instances of the HTML viewer portlet in Vignette console using the typical procedures for doing so.

    Note: on the Details tab for the portlet in Vignette console, you will be prompted for several portlet configuration preferences, including a default view filename and an optional, default includes filename. See Using the htmlviewer portlet below for detailed instructions. Be aware that, unless you have installed and configured i18n_portlet_config.properties (below) to point to a different location, the portlet will assume the view and include files are under /opt/sasuapps/sp/global_resources/portlet/i18n/. Also note that, until you configure a default view file, the portlet's view mode will display a generic internal error message (with error code error.internal) when no alternative view file is provided in the URL (see Linking to the portlet instance below).

1.1.2 Installation of i18n_portlet_config.properties (optional)

The HTML viewer portlet uses the SPF portlet utilities to find your view file and other local resources referenced from the view file (such as the includes file and any images, CSS, etc which you have locally stored). Therefore you may need to configure the SPF portlet utilities' i18n_portlet_config.properties, to tell the portlet where to search for those.

Note: You only need to concern yourself with i18n_portlet_config.properties if you use external view files for these local resources and you are putting the portlet resource bundle folder in a non-default location. (That is the external folder on the portlet server's filesystem, where you plan to put your externalized view content - see the discussion about external view files below.) If you are putting the portlet resource bundle folder in the default location (/opt/sasuapps/sp/global_resources/portlet/i18n) then you can skip this.

To install this file:

  • Download the template for i18n_portlet_config.properties. It is packaged with the rest of the SPF portlet configuration file templates, in the spf-portlet-config.jar. Download the latest version of this JAR (typically a version number is affixed to the JAR filename) and extract the i18n_portlet_config.properties template file from it.

    You can get the latest spf-portlet-config.jar from the SPF team on their Maven 2 repository, here: http://repo1.corp.hp.com/nexus/content/groups/public/com/hp/it/spf/xa/spf-portlet-config/ (browse for the latest version).

  • Customize the portlet.bundleDirectory property in the file, so that it has your non-default path for the portlet resource bundle folder. For example, if you plan to keep your external view files somewhere under /var/htmlviewer/content, then you would configure:

    portlet.bundleDirectory=/var/htmlviewer/content

    For more information about configuring this property, see the instructions both in the template file, and in the SPF portlet utilities guide (in the section about configuring the i18n_portlet_config.properties).

    Note: portlet.bundleDirectory is the only property in the file you will need to configure; ignore the rest.

  • Install the customized file, named i18n_portlet_config.properties, where the HTML viewer portlet can access it. You can put the file anywhere that the system classloader searches: you can add it into the htmlviewer.war itself (eg in /WEB-INF/classes) or you can put it in any folder outside of the WAR which is named in the classpath for your JVM. We recommend the latter; it makes administrative changes to the file easier, and also means that when a new htmlviewer.war is released, you can upgrade without re-inserting your file into it.

    If you are interested in externalizing the i18n_portlet_config.properties from the portlet WAR, check how your application server is configured. If you are using an application server hosting environment, like DASH or SASU, they probably will have provided you a "global resources" folder on the server filesystem and in the JVM classpath. Typically this is /opt/sasuapps/<your-program>/global_resources. You would put the i18n_portlet_config.properties file there.

1.1.3 Installation of init_relay.properties (optional)

The HTML viewer portlet uses the SPF portlet utilities' file relay servlet to host downloads of external static resource files (images, CSS, etc) which you are keeping in the portlet resource bundle folder and which are referenced from your view file. If your view file references such files, you may want to configure the SPF portlet utilities' init_relay.properties to configure the servlet to your satisfaction.

Note: You only need to concern yourself with init_relay.properties if you reference external resource files in your portlet resource bundle folder from your view file, and you are dis-satisfied with the configuration in init_relay_default.properties (which is delivered inside the SPF portlet utilities JAR). See the discussion in the package description for the relay servlet. For all of the most common static resources, the default configuration is fine and you should be able to skip this.

To install this file:

  • Download the template for init_relay.properties. It is packaged with the rest of the SPF portlet configuration file templates, in the spf-portlet-config.jar. Download the latest version of this JAR (typically a version number is affixed to the JAR filename) and extract the init_relay.properties template file from it.

    You can get the latest spf-portlet-config.jar from the SPF team on their Maven 2 repository, here: http://repo1.corp.hp.com/nexus/content/groups/public/com/hp/it/spf/xa/spf-portlet-config/ (browse for the latest version).

  • Customize the properties in the file as needed - see the instructions both in the template file, and in the SPF portlet utilities guide (in the section about configuring the init_relay.properties).

  • Include the customized file, named init_relay.properties, with your htmlviewer application. You can put the file anywhere that the system classloader searches. See the discussion on this point above, in regard to i18n_portlet_config.properties - the same comments apply to init_relay.properties.

1.1.4 Installation of html_viewer_includes.properties (optional)

The HTML viewer portlet uses the SPF file utilities' text interpolation engine to parse your view files and perform dynamic substitutions for certain special markup tokens you can use when authoring the view file. If you use the {INCLUDE:key} token, then you will need to configure an includes (aka token substitutions) property file to define your key-value pair(s). By default, this property file is named html_viewer_includes.properties.

Note: You only need to concern yourself with the includes property file if you use the special {INCLUDE:key} markup token in your view file. Otherwise you can skip this.

To install this file:

  • Download the template for the includes file from SPF team Maven 2 repository. The template file is named default_includes.properties but you will rename it. It is delivered inside the spf-portlet-config.jar file (often a version number is affixed to the end of the JAR file name). You can download that JAR file from the SPF Maven repository here: http://repo1.corp.hp.com/nexus/content/groups/public/com/hp/it/spf/xa/spf-portlet-config/ (browse for the latest version).

  • Add your key-value pairs to the file - see the instructions both in the template file, and in the SPF portlet utilities guide (in the section which discusses the {INCLUDE:key} token).

  • Include the customized file with your htmlviewer application. By default, you can name it html_viewer_includes.properties, but you can give it a custom name. (If you will have multiple portlet instances and you want them to use different property files, this is the way.) You have several choices where you can put this file:

    • You can put the file anywhere that the system classloader searches. See the discussion on this point above, in regard to i18n_portlet_config.properties - the same comments apply to your token substitutions property file.

    • Alternatively, you can put the file anywhere in the HTML viewer portlet WAR or in the portlet resource bundle folder, managing it as an internal or external resource file like the view file itself - see installing the view content.

    You can have multiple such include files. When you configure the portlet instance, you will tell it the filename of the one you want the portlet to use by default, and its location relative to where you put it. The portlet will then search in the above locations for it by default. Similarly, when you link to the portlet instance, you will have the option of overriding that default by passing the desired includes filename and location through the URL; the portlet will then search in the above locations for that.

1.2. Dependencies

All of the runtime dependencies for the HTML viewer portlet are contained inside the htmlviewer.war. Your application server should be using Java 1.5 minimum and your portlet container needs to support JSR-286.

2. Using the htmlviewer portlet

This chapter tells you how to use the HTML viewer portlet to render your content.

2.1. Authoring the view content

2.1.1. Authoring the view file

Your view file must be an HTML file with UTF-8 character encoding. In your HTML file you can use whatever text, HTML markup, CSS and JavaScript is desired. However, remember that the content of your file will be aggregated by the portal with other content to form the overall Web page. Therefore for best results, your view file should not include any of the following:

  • The HTML <html> and </html> tags.
  • The HTML <head> and </head> tags, including any markup valid only inside those tags (such as <meta> and <title>).
  • The HTML <body> and </body> tags.
  • File editors on the Microsoft platform often insert a "byte order mark" (BOM) in the first 2 bytes of a file, when the file is saved. To avoid any possible rendering problems, remove the byte order mark if it is present.

Your view file can contain any of the special markup tokens supported by the SPF portlet utilities' text interpolation engine. With these tokens, you can perform certain server-side substitutions such as:

  • Inserting user data like name, email, and locale;
  • Inserting URLs pointing to other portal pages at your portal site (though not targeting portlets with parameters, modes, window states, etc);
  • Inserting URLs pointing to other static resource files (CSS files, JavaScript files, images, video, audio, PDF, etc) contained outside of your view file - either inside the htmlviewer WAR itself, or outside it in the portlet resource bundle folder (see the discussion below);
  • Conditionally including content based on user authentication status or group- or role-based authorization; portlet ID or site; and date and time;
  • Including content kept as key-value pairs in a separate token substitutions property file (ie includes-file);
  • And other behaviors.

Please see the discussion of special markup tokens in the SPF portlet utilities guide for a full description of all of the dynamic server-side behaviors you can put into your HTML view file.

Note: Be advised that if you use the {INCLUDE:key} token, you will want to create an includes file (aka token substitutions file) to go with it. This is a simple property file where you will define the key-value pairs you reference from your view file. See the discussion about installing html_viewer_includes.properties above.

Note: Also be advised that the file relay servlet is already deployed inside the htmlviewer portlet application. You can use any of the tokens which build URLs to external static resources (images, etc) without having to worry about installing that servlet somewhere. (If you want to override that servlet's default configuration - usually this is not required - see the discussion above about installing init_relay.properties.)

When your view file is finished, give it any desired filename with either a .html or .htm file extension. If you plan to offer multiple localizations (translations) of that content, then create files with the same basename for each. Name the file containing the base content (ie the content localized for the base locale on your system) without any locale tag. Name the other files with a locale tag as per the Java standard. Translations of the same view file must have the same basename - ie disregarding the locale tag, their filenames must be the same. Like this:

my_view_file.html          # Base (eg English) view file
my_view_file_fr.html       # Generic French view file
my_view_file_fr_CA.html    # French (Canada) view file

Note: If you will be using the SPF localization business process, then typically you just write a base English view file, and the localized versions with the proper tagging will be created for you via that process. If you author your own localized versions, be sure to use the proper locale tag in your filename.

2.1.2. Other resources referenced from the view file

Your view file can include URLs pointing to other static resources: images, video, audio, PDF, CSS files, JavaScript files, etc. Author these files as you normally would for each of their respective formats. (For text files like CSS and JavaScript, you should use the same character encoding as the view file, namely UTF-8.) The special markup tokens cannot be used in these resources; they can only be used in the main HTML file (ie the view file) itself.

Name your resource file whatever you prefer, giving it a file extension appropriate for its format (eg .css for CSS, .js for JavaScript, .gif for GIF images, etc). Then store your resource files wherever you want, and reference them as usual from your view file. Or, store them as local resources - packaging them inside the HTML viewer WAR or outside the WAR in the portlet resource bundle folder, just as you would with the view file itself (see installing the view content below). To then reference these local resources from your view file, use the appropriate special markup token in your view file to make the URL pointing to the resource: either {CONTENT-URL:path} or {LOCALIZED-CONTENT-URL:path}. The path includes the filename as well as any subfolder where you plan to keep the resource file (see installing the view content below).

For example, if you have a local picture.jpg image which you plan to put into the images/ subfolder (of either the portlet resource bundle folder or the HTML viewer WAR itself), you could reference it from your view file like this:

<IMG SRC="{LOCALIZED-CONTENT-URL:/images/picture.jpg}" ALT="Some picture">

As with view files, resource files referenced through {LOCALIZED-CONTENT-URL:path} should be tagged with the locale they correspond to (except for base-locale files). For example, we would use picture.jpg if the image is English; picture_fr.jpg if it was generic French; picture_fr_CA.jpg if it was French for Canada; etc. (As with view files, though, if you will be using the SPF localization business process for your resource files, normally you will just create the base English versions, and the translators will create the localized version.)

2.2. Localizing the view content

Your view content - both your view file and any other local resources like images which you reference from your view file - can be localized. You can create the localized versions yourself if desired - just follow the procedures for proper character encoding and naming of the localized files as discussed in the previous section. Or, you can use the SPF-standard localization business process, in which you just create the English base file(s), and then send them off to be translated by the HP Translation and Localization (T&L) team. For more information about how this works, see the section about the SPF localization process in the SPF portlet utilities guide.

2.3. Installing the view content

Once your view content files are created - the main HTML view file(s) themselves, plus any local resource file(s) and include-files, as well as their localized versions - you have a choice as to where you put them. You can either add them to the htmlviewer.war itself, or you can put them outside, underneath a folder on the portlet server filesystem called the portlet resource bundle folder.

Note: This is the same principle as used in the SPF portlet utilities for custom portlet applications' message properties and other resource files: that is, internal versus external resources. The same ideas apply; you may want to read the discussion on where to put your resources in the SPF portlet utilities developer's guide. We recommend you put your content files outside of the htmlviewer.war; that way, when the SPF team provides an updated version of the WAR, you can upgrade to it without first adding your custom file.

Note: If you have an includes file, you can install it as an internal or external resource file, too. See the discussion about installing html_viewer_includes.properties above, and configuring the portlet instance below. As above, we recommend your put your includes file outside of the WAR.

Note: Unfortunately at this time the HTML viewer portlet does not support loading the view content or token substitution properties to a database; only to the portlet WAR or the filesystem. Similarly, at this time it does not provide a UI wizard or config mode for installing that content; you have to do it without the support of such a portlet-provided tool.

2.3.1. Packaging view content inside the htmlviewer WAR

You can choose to add your view content to the htmlviewer.war. The advantage of this is that now wherever the htmlviewer.war gets copied to, the content will go along with it. The disadvantage is that putting the content inside the WAR makes it more difficult for an administrator to change at runtime. Note also that if you put it inside the WAR, you will have to repeat that whenever you upgrade to a new version of htmlviewer.war from the SPF development team.

If you choose to package your view content inside the htmlviewer WAR, here is how:

  1. Explode the htmlviewer.war file as you normally would any WAR, EAR, JAR or ZIP file. Eg on Unix you can use jar xvf htmlviewer.war for this.
  2. For organizational purposes, create a desirable folder at the root of the exploded WAR, to store your view file(s). For example, create an html/ folder at the root of the exploded WAR.
  3. Copy your view file(s) - both the base file(s) and the localized version(s) if any - into the folder you just created (eg, the html/ folder). You can put your file(s) into that folder directly, or you can organize them into subfolders of that one. However, be sure to keep each localized bundle - ie each base file and its localized variants - together in the same folder.
  4. For each type of other local resource which you have, create an appropriate folder at the root of the exploded WAR, and likewise copy those resource files into it (again, both the base and localized versions if any). As with the view files, you can use further levels of subfolder to organize the static resources, so long as you keep all the files of each localized bundle together. Important: Make sure the relative path and file name (ie base name, in the case of a localized bundle of resource files) match what you used in your {CONTENT-URL:path} and {LOCALIZED-CONTENT-URL:path} references in your view file(s).
  5. Make sure all the folder and file permissions are set to allow read by the application server at runtime.
  6. Re-build the htmlviewer.war file as you normally would a WAR file. Eg on Unix you can use jar cvf htmlviewer.war * for this.

For example, imagine that we plan to have 2 HTML viewer portlet instances, each displaying different, localized views (including some shared images). One of the views uses a token substitutions file for the {INCLUDE:key} token. The layout could look like this:

<web-app-root>/
     html/
          my_portlet_instance_1/
               my_view_file_1.html
               my_view_file_1_de.html
               my_view_file_1_fr.html
               my_view_file_1_pt_BR.html
               ...
          my_portlet_instance_2/
               my_view_file_2.html
               my_view_file_2_de.html
               ...
     images/
          shared/
               an_image.gif
               an_image_de.gif
               an_image_fr.gif
               an_image_pt_BR.gif
               ...
     properties/
          my_portlet_instance_1/
               my_includes.properties
          ...

Note: The htmlviewer portlet WAR already contains many files. Be sure not to lose or corrupt any of them when inserting your content into the WAR or rebuilding the WAR.

2.3.2. Installing view content outside of the htmlviewer WAR

Alternatively, you can put your resource files outside of the htmlviewer.war, in the portlet resource bundle folder. This is a folder on the portlet server's filesystem which is devoted to storing external resource files for SPF-based portlet applications. By default, the HTML viewer portlet assumes the folder is /opt/sasuapps/sp/global_resources/portlet/i18n/ but you can override that in i18n_portlet_config.properties (see the discussion above about installing i18n_portlet_config.properties).

The advantage to externalizing your view content is that administrative changes are made easier than they would be if the content was packaged inside the WAR. Also, you can upgrade to a new version of the WAR without first re-packing your content inside it. However, you must also become concerned with distributing your view content and installing it into the right place in production, in addition to the WAR itself.

If you choose to use external view content, here is how:

  1. If you are using a non-default location for the portlet bundle folder, configure the location in i18n_portlet_config.properties and install that file as mentioned above.
  2. On each portlet server in your cluster:
    1. Create the portlet resource bundle folder if it does not already exist.
    2. Create a desirable folder underneath it for storing your view file(s), such as an html/ subfolder.
    3. Copy your view file(s) - both the base file(s) and the localized version(s) if any - into the folder you created (eg, the html/ folder). You can put your file(s) into that folder directly, or you can organize them into further subfolders. However, be sure to keep each localized bundle - ie each base file and its localized variants - together in the same folder.
    4. For each type of other local resource which you have, create an appropriate subfolder within the portlet bundle folder, and likewise copy those resource files into it (again, both the base and localized versions if any). As with the view files, you can use further levels of subfolder to organize the static resources, so long as you keep all the files of each localized bundle together. Important: Make sure the relative path and file name (ie base name, in the case of a localized bundle of resource files) match what you used in your {CONTENT-URL:path} and {LOCALIZED-CONTENT-URL:path} references in your view file(s).
    5. Make sure all the folder and file permissions are set to allow read by the application server at runtime.

Let's repeat the previous example, this time externalizing the content. The layout looks the same, except that it is relative to the portlet bundle directory (eg, /opt/sasuapps/sp/global_resources/portlet/i18n/ by default) instead of the WAR root:

/opt/sasuapps/sp/global_resources/portlet/i18n/

               my_view_file_1.html
               my_view_file_1_de.html
               my_view_file_1_fr.html
               my_view_file_1_pt_BR.html
               ...
          my_portlet_instance_2/

               my_view_file_2_de.html
               ...

               an_image_de.gif
               an_image_fr.gif
               an_image_pt_BR.gif
               ...

          my_portlet_instance_1/
               my_includes.properties

2.4. Creating and configuring the portlet instance

Once the htmlviewer application has been installed in your portlet application server, and your view content is prepared and installed, you are ready to instantiate and configure an instance of the portlet to display the content.

2.4.1. Creating the portlet instance

If your portlet deployment succeeded, and your portal is in communication with your portlet container, then in the Vignette console for your portal site, you will see the HTML viewer portlet listed as an available portlet type for instantiation. Just follow the standard procedures for creating an instance of a portlet type. You may name the portlet whatever you like, permission however you require, dress it in whatever portlet chrome of your choice as needed (or go chromeless), and position the portlet wherever needed on whatever page.

Note: If your view content uses the special {PORTLET:id}...{/PORTLET} markup token, then when you create the portlet instance for that content, be sure to enter a portlet friendly ID which matches that token. In Vignette console, the portlet friendly ID is entered on the Basics tab of the portlet instance.

The portlet friendly ID is also used in the logs, to differentiate between log entries for different portlet instances (see reading the logs below). So it is a good idea to specify a good friendly ID even if your view content does not use that token.

Note: Similarly, if your view content uses the special {PAGE:id}...{/PAGE} or {SITE:name}...{/SITE} markup tokens, then be sure to position the portlet instance on page(s) and portal site(s) using the particular page friendly ID or site DNS name. In Vignette console, the page friendly ID is entered on the Basics tab of the page, and the site DNS name is entered on the Basics tab for the site.

2.4.2. Configuring the portlet instance

After creating the portlet instance, go to its configuration view (ie run the portlet's config mode). This is where you tell the portlet instance what view file to render by default. In Vignette console, the configuration view is found in the Details tab of the portlet instance.

Note: Until you configure the portlet instance, its view mode will not know what view file to render by default, and will display an error if invoked without any view file parameter (see linking to your portlet instance, below). The config mode view will remind you to specify a default view file when you first enter the mode.

In the configuration view, there are 4 input elements:

Default view file:

In this text box, enter the name of the view file containing the content this portlet instance should render by default. In the case of a localized bundle of view files, enter the base name. Include the file extension, and any subfolders relative to the portlet bundle folder or portlet WAR root. Consider the example layout in the previous section:

  • When configuring the first portlet instance, you would enter just /html/my_view_file_1.html.
  • When configuring the second portlet instance, you would enter just /html/my_view_file_2.html.

Links to pages containing your portlet instance will be able to override this default view file using a parameter in the URL - read about linking to the portlet instance below.

Default includes file (ie token substitutions file):

Optional. You only care about this input element if your view file content uses the special {INCLUDE:key} token to dynamically insert key-value pairs from the includes file; and that property file is named something other than the default (html_viewer_includes.properties) or exists in a subfolder. If so, then enter the default filename here. Include the file extension, and any subfolder relative to the classloader folder, htmlviewer.war root, or portlet resource bundle folder where the file is located. (The file may be located in any folder searched by the system classloader, such as your "global resources" folder configured on the JVM classpath; or it may be anywhere inside the WAR or underneath the portlet resource bundle folder. See the discussion about installing html_viewer_includes.properties earlier in this guide.)

Consider the example layout in the previous section - for that example, you would enter /properties/my_portlet_instance_1/my_includes.properties.

Links to pages containing your portlet instance will be able to override this default includes file using a parameter in the URL - read about linking to the portlet instance below.

Open links in new buttonless window?

Optional. The HTML viewer portlet has a secondary feature whereby it can rewrite hyperlinks in your view content so that, when clicked, the link is opened in a new, buttonless, child window. Only hyperlinks coded in your view file using the <A HREF="..."> tag are rewritten. JavaScript must be enabled in the browser for the rewrite to have any effect (if JavaScript is disabled, the link still functions but does not launch a new buttonless window).

To enable this optional feature, click the checkmark in the configuration view. This feature is off by default.

Cache duration:

Optional. The HTML viewer portlet caches all of the resources it uses to render the view content - the view file, includes file, and the values of the these configuration input elements. This may improve the HTML viewer portlet's performance under heavy loads.

The cache duration input element lets you specify the number of seconds until the cache will be refreshed. By default, the cache duration is 900 seconds (15 minutes) but if you want to enter an alternative value, you can enter any positive number of seconds here. Enter a negative number (eg -1) to cache forever, and enter zero (0) to disable use of the cache.

Be sure to click the Save button to save your changes.

You are now done configuring the portlet, and you can go view or preview the portlet to see and test your rendered content.

  • Come back to the configuration view at any time to make changes (eg point the portlet instance at a different default view file).
  • You can also make changes to the view content at any time. Changes you make to external view content take effect immediately, without any restart or reconfiguration of the portlet required.

2.5. Linking to the portlet instance

By default, when a portal page containing the HTML viewer portlet instance is requested, the content from the default view (and includes) files will be rendered by that instance. However the URL used in the request can include parameters to override either or both of those resources with any alternative view (or includes) file on the system. When you make links pointing to the page, you can set these parameters in your link URL in order to force non-default views from the portlet.

The alternative view and includes filenames are specified through parameters in the URL named spfViewFile and spfIncludesFile, respectively. If you do not provide a parameter, then the default resource you entered when configuring the portlet instance will be used. But if you do provide a parameter, that resource will be used. If the resource is not found, the HTML viewer portlet will render an error message.

Parameters are provided in the query string of the portal page URL. Parameters may be provided in the URL in either of two ways: as portlet render parameters, targeted to one specific HTML viewer instance on the page; or as portal query parameters, applying to all HTML viewer instance(s) on the page. A portlet render parameter overrides a portal page parameter, which in turn overrides the default configured value.

For example, the following URL causes any and all HTML viewer portlet instances on the page to render the /news/article22.html view file, while the includes file is allowed to default:

http://my_host.hp.com/portal/site/my_site/my_page?spfViewFile=/news/article22.html

2.6. Reading the logs

The HTML viewer portlet uses WPAP (Timber) application logging. Configuration for these logs is in logger.xml, inside the htmlviewer portlet WAR file. The out-of-the-box configuration generates business activity, performance, error and error trace logs from which you can see how the portlet has been behaving. All HTML viewer portlet instances in the consuming portal write to these same logs.

Besides the standard fields logged by Timber, the HTML viewer portlet adds application-specific context data to the Timber logs. This includes the SPF diagnostic ID (used for correlating log records for the same session or request across multiple SPF log files at the Web, portal, and producer layers of the architecture), the portlet friendly ID (allowing you to distinguish between portlet instances in the log files), and whether the data rendered came from the cache or the source. For example, here is a business log record (the record has been wrapped here for readability):

09/17/2009, 19:13:04 | 12532147846432971 | 12532147846432971 | 
epi:user.standard;654e687954b2df310152dd6cb053ce01 | 
257dc89c9d78647ad00bb72db053ce01-56a6ec2482b21250e00bb72db053ce01-2636b6e39487235532e82a10b053ce01 | 
RENDER_PHASE | view | normal |  | en_US | 
{diagID:2c08c1e8+12cd20e7757+06},{portletID:hmLogin},{contentFrom:cache} |  | OK | 0.008

This log record was for the hmLogin portlet instance (friendly ID) and indicates the content (US English) was served succesfully (OK) from cache in 8 milliseconds. The diagnostic ID is 2c08c1e8+12cd20e7757+06, indicating this log record was for the 6th portlet aggregated on the page in the 12cd20e7757 request in the 2c08c1e8 session; this information can be used to correlate multiple business log records with one another or with log records from the OpenPortal, Vignette, and Apache logs as well - providing a log trace through the whole SPF architectural stack.

Application errors like missing or unreadable view files are logged, as well. For example, when a portlet instance is requested to point to a non-existent view file, it has no content it can display. When a user views the portlet anyway, it will render a service-unavailable error message, and the business log entry will look like this:

09/17/2009, 19:19:21 | 12532151611575614 | 12532151611575614 | 
epi:user.standard;654e687954b2df310152dd6cb053ce01 | 
257dc89c9d78647ad00bb72db053ce01-671d9ad3f4880aa391c6b510b053ce01-eae66195bd3b616d45c1fbecb053ce01 | 
RENDER_PHASE | view | normal |  | en_US | 
{diagID:1c44b00d+12cd1ad5638+01},{portletID:fmHelp},{contentFrom:source},
{error.nullFile:viewFilename not found or readable: forums/html/forumsHelp1.html} |  | FATAL | 0.061

This log record was for the fmHelp portlet instance (friendly ID) and indicates there was a FATAL error due to the view file (forums/html/forumsHelp1.html) not being found. The error and error trace logs will contain further information.

The out-of-the-box HTML viewer configuration puts these log files into /opt/sasuapps/sp/logs/htmlviewer/main (where htmlviewer is the context root for the application, and /opt/sasuapps/sp/logs/ is the WPAP Timber log root defined in the HTML viewer application's web.xml).

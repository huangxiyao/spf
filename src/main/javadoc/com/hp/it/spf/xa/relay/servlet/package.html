This package contains Java classes used exclusively by and for the SPF file relay servlet - the servlet which hosts downloads of external supporting resource files from the portlet resource bundle folder. SPF portlet developers generally will not use these classes directly. However, if they are going to have external supporting resources, they will need to properly configure and deploy this servlet. The steps are as follows:

1. Configuring web.xml

The relay servlet class is {@link RelayServlet} and it is contained inside the SPF portlet utilities JAR. You need to add a deployment descriptor for the servlet into your application's web.xml file. The deployment descriptor just looks like this:

	
		FileRelayServlet
		
			com.hp.it.spf.xa.relay.servlet.RelayServlet
		
	
	
		FileRelayServlet
		
		/relay/*

As noted in the above comment, you can use a different path for the relay servlet mapping. But we recommend you use /relay (ie the current value of {@link com.hp.it.spf.xa.i18n.portlet.I18nUtility.RELAY_SERVLET_PATH}) since that is the default path for the relay servlet. If you provide a different one, you will have to do extra configuration in i18n_portlet_config.properties (see below).

2. Configuring init_relay.properties (optional)

The relay servlet only lets users access files with certain permitted file extensions. The permitted file extensions are listed in a properties file, init_relay.properties. For each listed file extension, a MIME type is also listed - this is the value which the relay servlet will use for the Content-Type: HTTP response header during the download.

A file inside the SPF portlet utilities JAR, init_relay_default.properties, defines a default set of these file extensions and MIME types. You should find that most general-purpose file extensions are listed there (for popular image formats like GIF and JPEG, popular video formats like MPEG, plain and enhanced text like HTML, MS Word and PDF, etc). And note that the text formats are assumed to be UTF-8 encoded, so the init_relay_default.properties specifies ;charset=utf-8 on their MIME types accordingly.

If you are happy with this set of relay servlet properties, then you can skip this step - there is nothing you need to do.

If you need to override the init_relay_default.properties, you can get a template for init_relay.properties from TBD. Update the properties in the template using the following format:

contentType.file-extension=mime-type

For example:

contentType.mov=video/quicktime # For Apple quicktime format using .mov extension

Then deploy your init_relay.properties inside your application, where the classloader will find it. This will override init_relay_default.properties in its entirety. See the deployment discussion below.

3. Configuring i18n_portlet_config.properties (optional)

The relay servlet needs to know the location of the portlet resource bundle folder, unless you are using the default location {@link com.hp.it.spf.xa.i18n.portlet.I18nUtility#BUNDLE_DIR_DEFAULT}. Also, if you mapped it under anything but the {@link com.hp.it.spf.xa.i18n.portlet.I18nUtility.RELAY_SERVLET_PATH} path, the SPF portlet utilities need to know that (for example, so that {@link com.hp.it.spf.xa.i18n.portlet.I18nUtility#getLocalizedFileURL(PortletRequest,PortletResponse,String)} can build a proper URL pointing at it).

Note: So if you use the default location for the portlet bundle folder and the relay servlet path, you can skip configuring i18n_portlet_config.properties.

You can get a default copy of i18n_portlet_config.properties from TBD. Follow the instructions in the property file to set the portlet bundle folder and relay servlet path as needed. (You can also see the instructions in the SPF Portlet Utilities Developer's Guide.)

Then deploy your i18n_portlet_config.properties inside your application, where the classloader will find it. See the deployment discussion below.

4. Deploying the servlet

We recommend including the relay servlet as part of your portlet application. In other words, you would do all the above steps for web.xml, i18n_portlet_config.properties, and init_relay.properties inside your portlet WAR. (The relay servlet will function as a portlet resource, over WSRP and through the portal as with other portlet application resources. Relay servlet URLs returned from the I18nUtility will already be portlet-encoded for this.)

However you can deploy the relay servlet inside a different Web application instead. The requirements are:

  • The relay servlet Web application must have access to the same portlet resource bundle folder (and all the supporting resource files therein) that your portlet application does.
  • You must configure the new, non-default base relay servlet URL in i18n_portlet_config.properties. (And if your portlet bundle folder is also non-default, configure that there too.)
  • You must provide identical copies of that i18n_portlet_config.properties to your portlet application and the relay servlet application.
  • Finally, your relay servlet Web application must be accessible to the user browser via the base relay servlet URL which you configured in i18n_portlet_config.properties.

Whichever application you deploy the servlet in:

  • Put the SPF portlet utilities JAR in the /WEB-INF/lib/
  • If you are providing init_relay.properties and/or i18n_portlet_config.properties, put that somewhere the system classloader can find it (eg, inside your WAR in the /WEB-INF/classes folder - or you can put them outside, as external resources, in the "global resources" folder configured on your JVM classpath).
